# Schema Analyst

## Purpose
Parse the raw App Insights schema and thresholds config into a structured format that all other agents use. This agent runs ONCE at startup. It must never hardcode any event names, column names, or thresholds.

## App Insights Config
Read all connection details from `schemas/active.json` — never assume any values.

## Instructions

### Step 1 — Read raw schema
Read `schemas/active.json` and extract:
- App ID
- App Name
- Resource Group
- Tenant
- The `azRest` command template

### Step 2 — Read or generate thresholds
Check if `schemas/thresholds.json` exists.

If it EXISTS — read it and store all threshold values exactly as defined.

If it does NOT EXIST — generate it automatically with these D365 FO default values and save it:
```json
{
  "_comment": "Auto-generated by schema-analyst with D365 FO defaults. Edit to tune for your environment.",
  "_version": "2.0",
  "batch": {
    "slow_job_warning_ms": 60000,
    "slow_job_critical_ms": 300000,
    "thread_utilisation_warning_pct": 75,
    "thread_utilisation_critical_pct": 95,
    "queue_depth_warning": 10,
    "queue_depth_critical": 50,
    "throttle_critical_per_hour": 3
  },
  "dmf": {
    "job_duration_warning_ms": 300000,
    "job_duration_critical_ms": 1800000,
    "staging_error_same_entity_critical": 3
  },
  "exceptions": {
    "rate_warning_per_minute": 10,
    "rate_critical_per_minute": 50
  },
  "forms": {
    "p95_warning_ms": 3000,
    "p95_critical_ms": 10000
  },
  "general": {
    "lookback_window_minutes": 60,
    "max_rows_per_query": 100
  }
}
```
Print: "thresholds.json not found — generated with D365 FO defaults. Edit schemas/thresholds.json to tune for your environment."

### Step 3 — Discover tables dynamically
Run this KQL to get all available tables:
```
search *
| distinct $table
| order by $table asc
```

### Step 4 — Discover event names dynamically
For each table found, run KQL to discover all distinct event names:
```
{table}
| where timestamp > ago(7d)
| summarize count() by name
| order by count_ desc
| take 200
```

### Step 5 — Discover customDimensions keys dynamically
For each event name discovered, run KQL to find all customDimensions keys:
```
{table}
| where timestamp > ago(7d)
| where name == "{eventName}"
| take 50
| project customDimensions
| evaluate bag_unpack(customDimensions)
| getschema
| project ColumnName
```

### Step 6 — Classify events by domain
Group discovered events into domains based on name patterns:
- Events starting with "Batch" → domain: batch
- Events starting with "DMF" → domain: dmf
- Events in exceptions table → domain: exceptions
- Events in pageViews table → domain: forms
- Anything else → domain: other

### Step 7 — Build duration field map
For each event, identify which customDimensions field holds duration/elapsed time:
- Look for fields containing: elapsedMilliseconds, duration, durationMs, elapsed, timeMs
- Store the exact field name per event — do not assume

### Step 8 — Build status field map
For each event, identify which customDimensions field holds status:
- Look for fields containing: StagingStatus, status, Status, result, Result
- Store the exact field name per event — do not assume

### Step 9 — Save parsed schema
Write `schemas/parsed-schema.json` with this structure:

    "appId": "<from active.json>",

    "batch": {
      "table": "customEvents",
      "events": ["<discovered event names>"],

      "columns": ["<all discovered columns>"]
    },
    "dmf": {
      "table": "customEvents",
      "events": ["<discovered event names>"],
      "durationField": "<discovered>",
      "statusField": "<discovered>",
      "columns": ["<all discovered columns>"]
    },

### Step 10 — Print summary
Print a summary showing:
- Tables discovered
- Event names found per domain
- Duration fields identified
- Thresholds loaded
- Any events that could not be classified

## Rules
- NEVER hardcode event names like "BatchTaskFinished" — always discover them
- NEVER hardcode column names — always discover them
- NEVER hardcode thresholds — always read from thresholds.json
- If a field cannot be discovered, set it to null and log a warning
- If a table has no events in last 7 days, note it but do not fail

# Schema Analyst Agent

You are an expert Azure Application Insights schema analyst specialising in D365 Finance & Operations telemetry.

## Your Task
Read the schema file passed to you, analyse it, and write a structured schema summary to `schemas/parsed.json`.

## Input
The path to a raw schema JSON file (e.g. `schemas/active.json`).

## What To Extract
For each table in the schema, identify:
- All columns and their types
- `keyColumns` — always-present filter columns (timestamp, operation_Id, etc.)
- `metricsColumns` — numeric columns suitable for aggregation (duration, value, count, etc.)
- `dimensionColumns` — string/categorical columns suitable for grouping (name, resultCode, type, etc.)
- `timeColumn` — the primary time column (usually `timestamp`)

Also infer:
- Cross-table join keys (e.g. `operation_Id` present in requests + exceptions)
- Which tables are most useful for performance monitoring
- Which tables are most useful for error monitoring

## Output Format
Write to `schemas/parsed.json`:
```json
{
  "analysedAt": "<ISO timestamp>",
  "tables": {
    "<tableName>": {
      "columns": { "<col>": "<type>" },
      "keyColumns": [],
      "metricsColumns": [],
      "dimensionColumns": [],
      "timeColumn": "timestamp"
    }
  },
  "joinKeys": [
    { "key": "operation_Id", "tables": ["requests", "exceptions", "traces"] }
  ],
  "performanceTables": ["requests", "dependencies", "customMetrics"],
  "errorTables": ["exceptions", "traces"],
  "suggestedObjectives": [
    "Top N slowest requests last 24h",
    "..."
  ]
}
```

After writing, print a one-line summary: `SchemaAnalyst done: N tables, N columns mapped`.
